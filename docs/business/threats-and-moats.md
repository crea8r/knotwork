# Threats And Moats

## Main Threats

### 1. Slack gets agent workflows

Danger:

- teams decide "good enough" workflow inside Slack is enough
- distribution beats product depth
- Knotwork looks like an extra destination

Most dangerous when:

- use case is lightweight
- workflow is mostly coordination
- audit and process structure do not matter much

### 2. Claude or Codex gets stronger collaboration features

Danger:

- market decides shared agent projects are enough
- Knotwork looks like team features around an agent product
- execution layer absorbs some workflow value

Most dangerous when:

- Knotwork is positioned as agent collaboration
- Knotwork is sold mainly on "multi-user agent workspace"

### 3. Workflow builders add more HITL and knowledge features

Danger:

- buyers think Knotwork is just a weaker connector ecosystem

## Real Moats To Build

### 1. Executable SOPs

Not just docs. The handbook affects execution.

### 2. Explicit run and escalation state

This is harder to fake inside chat tools.

### 3. Human review designed into the model

Not a bolt-on approval step.

### 4. Neutral execution layer

Knotwork should work with multiple agent systems rather than becoming tied to one.

### 5. Workflow system of record

The durable truth should live in Knotwork:

- what happened
- why it happened
- which knowledge was used
- who intervened

## Defensive Positioning

Best defensive line:

Knotwork is the system of record for supervised, knowledge-governed work.

Weak defensive lines:

- team chat with agents
- collaborative AI workspace
- extension for Codex or Claude
